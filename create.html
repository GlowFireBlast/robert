Development Process and Key Milestones

The development of my Camping Environment Assistant followed a structured and fairly straightforward process:

  • The Initial Prototype: I began by programming the Micro:bit to successfully read analog sensor data (temperature, light level, and sound level). I also implemented basic digital button controls (A, B, A+B) to trigger different functions.

  • Data Logging Implementation: Once sensor readings were reliable, I developed a system to log the data into a CSV file. This involved formatting the data correctly (time, light, temperature, sound) and ensuring consistency for later processing.

  • Python Program Dev: I created this program to: Read and process CSV data, compare logged environmental conditions, incorporate user preferences and determine the most suitable camping conditions.

  • "What-if" Scenario Integration: I implemented two "what-if" scenarios (e.g. temperature increase due to drought), allowing the system to simulate environmental changes and show how they affect optimal camping conditions.


Testing

Testing was continuous and took place at each stage:

  • Sensor Testing: I tested each sensor individually to ensure they all had accurate readings. For example, I exposed the Micro:bit to different light levels and temperatures to confirm that values changed appropriately.

  • Input Testing: I verified that each button and button combination triggered the correct function (e.g, Buttons A+B -> SOS Alert and Buzzer)

  • User Testing: I conducted testing with my own friends and family, all potential users, to gather feedback on usability, accessibility and functionality, making slight, but well needed adjustments based on their input.

Problems and Solutions

One significant problem that I encountered was during the development of the Micro:bit system. The values that were being logged were inconsistent and often incorrect.

  • Cause: After troubleshooting, I discovered that the issue was due to error in how often the sensors logged data. I had set the logging interval too short, which caused the sensors to read and log data before they had stabilized, leading to inaccurate readings.

  • Solution: To solve this, I adjusted the programme to log data based on button inputs rather than a fixed time interval, allowing for more accurate and consistent data collection. This also made the system more user-friendly, as users could choose when to log data based on their current environment.

Model/Algorithm Description

The core of my system is an algorithm that evaluates environmental suitability for camping based on sensor data and user preferences.

  1. Input Collection: The system reads the temperature, light level and sound level from the CSV file.

  2. Preference Matching: The algorithm compares the current conditions against user-defined preferences (e.g. ideal temperature range, preferred light level).

  3. Scoring System: Each environment reading is scored based on how closely it matches the users preferences:
    • Temperature within preferred range -> higher ranking
    • Light level within preferred range -> higher ranking
    • Sound level within preferred range -> higher ranking

  4. Risk Detection: The sstem checks for potentially dangerous conditions (High temperature -> forest fire risk) and if detected, it triggers an alert and provides safety recommendations.

  5. Optimal Environment Selection: The system compares all recorded data points and selects the one closest to the user's preferences as the optimal camping environment.

  6. "What-if" Scenarios: The algorithm modifies input data to simulate changes such as "What if temperatures increase by 5°C due to drought conditions?". It then outputs the new most optimal environment and detects the same risks assessed beforehand.

Overall, This implementation demonstrates both the embedded system (Micro:bit) and the adaptive system (Python) working together to solve a real world problem, while also incorporating testing and problem-solving.
